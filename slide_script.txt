SLIDE 2 - THE 3 RESEARCH QUESTIONS
----------------------------------------------------------------
Show the 3 RQs side by side or stacked.

RQ1: How do building codes and standards affect the production,
     rehabilitation, and affordability of housing?

RQ2: Do codes and standards better protect homes and communities
     from climate-related risks such as hurricanes, wildfires,
     sea-level rise, and extreme heat?

RQ3: What are the impacts of more stringent code adoption
     on housing vulnerable communities?

SCRIPT:
"Everything we build has to trace back to these three questions.

RQ1 asks how building codes affect housing production,
affordability, and rehabilitation.

RQ2 asks whether codes actually protect communities from
climate risks -- wildfire, hurricane, flooding, extreme heat.

RQ3 asks what happens to vulnerable communities when codes
get stricter.

The challenge is that answering these questions requires
connecting building code text -- which is unstructured,
scattered across hundreds of PDFs -- to measurable outcomes.
That is exactly what our pipeline does."


----------------------------------------------------------------
SLIDE 3 - THE CORE PROBLEM
----------------------------------------------------------------
Show: 381 PDFs | Manual search | No structure

SCRIPT:
"Without infrastructure, a researcher trying to answer RQ2
would manually open dozens of PDFs across multiple cities and
edition years, search for wildfire-related sections, and try
to compare them by hand. That does not scale to 121 cities.

Our answer is a GraphRAG system -- combining a knowledge graph
in Neo4j with a vector database in Supabase -- so that a
researcher can ask a natural language question and get
structured, citable answers back."


----------------------------------------------------------------
SLIDE 4 - ARCHITECTURE OVERVIEW
----------------------------------------------------------------
Show diagram:
  Shared OneDrive
      |
      v
  Scanner (scan.py)
      |
      v
  SQLite Manifest  -->  Parser (parse.py)
                              |
                    ----------+----------
                    |                   |
                    v                   v
                 Neo4j               Supabase
              (Graph DB)           (Vector DB)
                    |                   |
                    +--------+----------+
                             |
                             v
                      Streamlit Chatbot

SCRIPT:
"Here is the pipeline at a high level.

First, a scanner reads the shared OneDrive folder and builds
a manifest -- a SQLite database that tracks every file, its
metadata, and its ingestion status.

Second, a parser reads each PDF and extracts structured
content. Chapters become DocumentUnit nodes. Sections become
CodeUnit nodes.

Third, those nodes are pushed into Neo4j as a knowledge graph,
and the text is embedded and stored in Supabase for semantic
search.

Finally, the Streamlit chatbot uses both sources together --
graph for structure, vectors for meaning -- to answer
researcher questions."


----------------------------------------------------------------
SLIDE 5 - NEO4J GRAPH SCHEMA
----------------------------------------------------------------
Show: 12 node labels and key relationships as a diagram.

Node Labels:
  State, Jurisdiction, Hazard, CodeFamily, CodeEdition,
  SourceFile, DocumentUnit, CodeUnit, Clause,
  AmendmentEvent, Chunk, Fingerprint

Key Relationships:
  State -[HAS_JURISDICTION]-> Jurisdiction
  Jurisdiction -[AFFECTS_HAZARD]-> Hazard
  Jurisdiction -[HAS_CODE_EDITION]-> CodeEdition
  CodeEdition -[HAS_DOCUMENT_UNIT]-> DocumentUnit
  DocumentUnit -[HAS_CODE_UNIT]-> CodeUnit
  Jurisdiction -[AMENDMENT_EVENT]-> AmendmentEvent

SCRIPT:
"The schema has 12 node types. For this pilot the most
critical ones are:

State and Jurisdiction -- representing California,
Los Angeles, and San Diego.

Hazard -- seeded with six FEMA hazard types: wildfire,
hurricane, flood, seismic, extreme heat, sea level rise.

CodeFamily and CodeEdition -- representing the Building,
Residential, and Fire code families and their specific
year editions.

SourceFile, DocumentUnit, CodeUnit -- representing the
actual PDFs, their chapters, and their individual sections.

And AmendmentEvent -- which is our treatment variable for
RQ2 and RQ3, tracking when each city adopted or amended
a code edition.

These nodes are connected by 29 relationship types."


----------------------------------------------------------------
SLIDE 6 - HOW THE SCHEMA ANSWERS EACH RQ
----------------------------------------------------------------
Show a 3-row table:

RQ   | Key Graph Path                              | Key Data
-----|---------------------------------------------|---------------------------
RQ1  | Jurisdiction -> CodeEdition -> CodeUnit     | Density, ADU, setbacks
RQ2  | Jurisdiction -> AFFECTS_HAZARD -> Hazard    | WFIR loss $, AmendmentEvent
     | CodeUnit -> AFFECTS_HAZARD -> Hazard        | treatment year
RQ3  | Jurisdiction -> AmendmentEvent              | Code stringency x SES
     | + population / income enrichment            | comparison

SCRIPT:
"Each research question maps to a specific traversal path
in the graph.

For RQ1, we walk from a Jurisdiction through its code editions
to find the CodeUnit sections that govern housing -- setbacks,
density limits, ADU requirements -- and compare them across
cities and years.

For RQ2, the key path is Jurisdiction to Hazard. Each pilot
jurisdiction is tagged with its wildfire exposure in dollars:
Los Angeles at 31 million, San Diego at 16 million. They are
linked to AmendmentEvent nodes that tell us whether and when
they adopted more protective codes. That is the treatment
variable for the difference-in-differences analysis.

For RQ3, we will enrich Jurisdiction nodes with Census income
and poverty data, then compare amendment adoption rates
across income groups."

SLIDE 9 - THE RAG LAYER
----------------------------------------------------------------
Show flow diagram:
  Researcher types question
        |
        v
  Embedding model converts to vector
        |
        v
  Supabase: top-5 relevant code sections
        |
        v
  Neo4j: add structural context (chapter, edition, city)
        |
        v
  LLM synthesizes answer with citations
        |
        v
  Response with section numbers + page ranges

SCRIPT:
"The graph alone is not enough for researchers who want to
ask natural language questions. That is where the RAG
layer comes in.

A researcher types: 'What wildfire provisions does LA's
2022 Fire Code include?'

The system converts that to an embedding, searches Supabase
for the most relevant code sections, then traverses the
Neo4j graph to add structural context -- which chapter,
which edition, which jurisdiction.

The LLM then synthesizes a factually grounded answer with
citations back to the exact section numbers and page ranges.
No hallucination -- every claim is traceable to a specific
CodeUnit node in the graph."
